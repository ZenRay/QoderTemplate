# 知识材料管理方案

> **定位**：在 AI 辅助编程实践过程中，系统性地捕获、沉淀和精炼会话中产生的重要内容，
> 为后续个人知识库建设提供高质量原材料。

---

## 决策结论

### 方案 A：跟随项目存储

```
{project}/.qoder/session-notes/
└── 2026-04-30_主题.md
```

- **存储位置**：各项目的 `.qoder/session-notes/` 目录下，随项目代码管理
- **草稿产生后**：直接保存在当前项目内，可随 Git 提交
- **精炼机制**：无统一归档，内容留在各项目中
- **适用场景**：内容强绑定单一项目、有团队共享需求

| 优点 | 缺点 |
|------|------|
| 与代码上下文强关联，可提交 Git | 内容分散在各项目，换项目后无法检索 |
| 落地极简，零配置 | 无法形成跨项目的个人知识沉淀 |
| 团队可共享 | 长期累积后碎片化严重 |

---

### 方案 B：用户级全局存储

```
~/.qoder/knowledge/
└── archive/2026/04/
    └── 2026-04-30_ProjectA_主题.md
```

- **存储位置**：用户 Home 目录下的 `~/.qoder/knowledge/`，所有项目共用
- **草稿产生后**：直接写入全局目录，不经过项目内过渡
- **精炼机制**：内容直接归档，依赖用户自律维护质量
- **适用场景**：不需要与代码关联、纯个人知识沉淀

| 优点 | 缺点 |
|------|------|
| 天然跨项目集中，全局可搜 | 与代码上下文脱节，缺少项目背景 |
| 长期维护成本低 | 不便于团队共享 |
| 个人知识沉淀强 | 草稿与精炼内容混在一起，质量参差 |

---

### 方案 C：双层结构（已采用）✅

```
实践过程（Qoder 会话）
       │
       ▼
草稿层：{project}/.qoder/notes/        ← 即时捕获，成本极低（不提交 Git）
       │  定期精炼（KnowledgeExtractor Skill）
       ▼
精炼层：~/Documents/PersonalKnowledge/  ← 跨项目统一归档，个人知识库原材料
```

- **草稿层**：项目内 `.qoder/notes/`，随手记录，不提交 Git，格式自由
- **精炼层**：`~/Documents/PersonalKnowledge/`，经过 KnowledgeExtractor Skill 提炼后统一归档
- **触发机制**：`PreCompact` / `SessionEnd` Hook 自动提示，或用户手动触发
- **适用场景**：兼顾即时捕获与长期沉淀，个人知识管理的推荐方案

| 优点 | 缺点 |
|------|------|
| 草稿零摩擦，想记即记 | 需维护两个位置 |
| 精炼后跨项目集中归档 | 初期需要建立精炼习惯 |
| 草稿求全、精炼求精，职责分明 | 配置成本略高于方案 A/B |

**选择方案 C 的核心理由：**
1. 草稿在项目内产生，零摩擦——有想法立刻记，不需要切换目录
2. 精炼内容统一归档，跨项目可检索——换项目后知识不流失
3. 两个阶段职责清晰：草稿求全，精炼求精

---

### 方案 D：独立知识 Git 仓库

```
~/Documents/PersonalKnowledge/   ← 独立 Git 仓库
├── sessions/
├── topics/
└── projects/
```

- **存储位置**：独立维护的 Git 仓库，与所有项目代码隔离
- **草稿产生后**：直接写入该仓库，可推送到私有远端（GitHub/GitLab）
- **精炼机制**：完整的版本控制，可追踪内容演变历史
- **适用场景**：知识库规模较大、需要版本管理或多设备同步时

| 优点 | 缺点 |
|------|------|
| 完整 Git 版本管理，可追溯 | 需额外维护一个仓库 |
| 可推送到云端，多设备同步 | 与代码上下文完全脱节 |
| 可配置为团队私有知识库 | 初始配置成本最高 |

> 方案 D 是方案 C 精炼层的**自然演进终态**——当 `~/Documents/PersonalKnowledge/` 内容积累足够多时，对其执行 `git init` 即可无缝升级为方案 D。

## 目录结构

### 草稿层（项目内，不提交 Git）

```
{project}/.qoder/notes/
├── 2026-04-30_Qoder工程化配置规范.md
├── 2026-05-02_API接口设计讨论.md
└── 2026-05-05_性能优化方案对比.md
```

**特点**：
- 随会话产生，可以很粗糙
- 加入 `.gitignore`，纯个人材料
- 不需要立即整理，积累后批量精炼

### 精炼层（全局，长期维护）

```
~/Documents/PersonalKnowledge/
├── archive/                        ← 精炼后的原始归档（按时间）
│   └── 2026/
│       ├── 04/
│       │   ├── 2026-04-30_QoderHarness_工程化配置规范.md
│       │   └── 2026-04-28_ProjectA_API设计决策.md
│       └── 05/
│
├── topics/                         ← 按主题聚合（便于专题复盘）
│   ├── engineering/                # 工程化实践、配置、工具链
│   ├── architecture/               # 系统设计、架构决策
│   ├── decision-records/           # 重要技术决策（ADR 风格）
│   ├── performance/                # 性能分析与优化
│   ├── security/                   # 安全相关
│   └── devops/                     # 部署、自动化、运维
│
├── projects/                       ← 按项目聚合（按项目复盘）
│   └── {ProjectName}/
│       └── index.md                # 项目知识总览与索引
│
└── README.md                       ← 全局总索引与导航
```

## 精炼内容更新方案

### 触发时机

| 时机 | 方式 | 触发机制 |
|------|------|----------|
| 会话上下文压缩前 | 自动提示 | `PreCompact` Hook |
| 会话即将结束时 | 自动提示 | `SessionEnd` Hook |
| 重要决策完成后 | 手动触发 | 用户直接要求 Qoder 执行 |
| 定期整理 | 手动触发 | 每周/每月批量处理草稿 |

### 草稿 → 精炼的完整流程

```
┌─────────────────────────────────────────────────────────────┐
│  阶段一：草稿捕获（零摩擦，会话中随时进行）                      │
│                                                             │
│  会话进行中                                                  │
│  ──────────                                                 │
│  用户/Agent 随手记录 → .qoder/notes/YYYY-MM-DD_主题.md       │
│  • 可以是 bullet points                                      │
│  • 可以是未经整理的讨论片段                                    │
│  • 可以是代码片段 + 简短说明                                   │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           │  PreCompact 或 SessionEnd 触发提示
                           │  或用户手动触发
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段二：精炼提取（KnowledgeExtractor Skill 执行）            │
│                                                             │
│  Agent 梳理当前会话内容：                                     │
│  1. 提取会话目标（一句话概述）                                  │
│  2. 识别核心决策和方案对比                                     │
│  3. 整理背景信息和约束条件                                     │
│  4. 汇总后续行动项                                            │
│  5. 生成结构化 Markdown                                      │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段三：归档写入                                             │
│                                                             │
│  写入路径：~/Documents/PersonalKnowledge/archive/YYYY/MM/    │
│  文件名：YYYY-MM-DD_ProjectTag_主题标题.md                    │
│                                                             │
│  同步更新：                                                  │
│  • topics/ 下对应主题目录（软链接或副本）                       │
│  • projects/{ProjectName}/index.md（追加条目）                │
│  • README.md（更新"最近归档"列表）                            │
└─────────────────────────────────────────────────────────────┘
```

### 精炼内容结构化模板

精炼后写入 `~/Documents/PersonalKnowledge/archive/` 的标准格式：

```markdown

project: ProjectTag
topics: [engineering, architecture]
date: YYYY-MM-DD
session_summary: 一句话描述本次会话目标

> **概述**：[一句话总结，复盘时快速判断是否需要深读]

## 背景信息

- 项目背景：...
- 技术约束：...
- 涉及模块：...

### 决策：[决策主题]

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 方案 A | ... | ... | ... |
| 方案 B | ... | ... | ... |

**选择**：方案 X  
**理由**：...

## 结论与建议

- 推荐做法：...
- 注意事项：...
- 可复用的模式：...

## 后续行动

- [ ] ...
- [ ] ...

- 代码文件：`src/...`
- 相关决策：[链接]

---
*精炼于 YYYY-MM-DD | 来源：Qoder 会话*
```

### 文件命名规范

```
YYYY-MM-DD_ProjectTag_主题标题.md

示例：
2026-04-30_QoderHarness_工程化配置规范.md
2026-05-02_ProjectA_API接口设计决策.md
2026-05-05_通用_Git提交规范最佳实践.md  ← 不特定于项目时用"通用"
```

**命名规则：**
- 日期：内容产生当天
- ProjectTag：3~10 字符，驼峰或中文均可；跨项目通用内容使用 `通用`
- 主题标题：10~30 字符，避免特殊符号

## 渐进式使用路径

### 第一阶段：手动草稿（立即可用）

```bash
# 会话中随时创建草稿
# 文件位置：{project}/.qoder/notes/
```

- 不需要任何自动化，想记就记
- 格式自由，bullet point 即可
- 积累 5~10 篇后进入第二阶段

### 第二阶段：Hook 提示 + 手动精炼

- `PreCompact` / `SessionEnd` Hook 自动提示
- 用户决定是否执行 KnowledgeExtractor Skill
- Agent 梳理并写入 `~/Documents/PersonalKnowledge/`

### 第三阶段：定期主题汇总

- 每月将 `archive/` 下同主题内容汇总到 `topics/` 专题文档
- 形成可独立阅读的"专题综述"
- 这是知识库建设的最终形态

## 快速操作参考

### 创建草稿

直接在会话中告诉 Qoder：

> "把刚才关于 XX 的讨论要点记录到 `.qoder/notes/` 下"

### 触发精炼归档

> "执行 KnowledgeExtractor Skill，将本次会话的重要内容归档到个人知识库"

### 查看知识库

```bash
# 查看最近归档
ls -lt ~/Documents/PersonalKnowledge/archive/2026/

# 全文搜索
grep -r "关键词" ~/Documents/PersonalKnowledge/

# 查看总索引
open ~/Documents/PersonalKnowledge/README.md
```
